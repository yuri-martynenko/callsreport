# Модель оценки звонка

## Назначение документа

Этот документ описывает, как в приложении формируется оценка разговора по ключевым показателям:

- сентимент;
- риск;
- score;
- соблюдение скрипта.

Документ фиксирует текущую логику системы в коде и отдельно показывает, что уже настраивается через сценарий, а что пока не вынесено в отдельные настройки.

## Общий принцип

Оценка звонка в текущей версии строится в два этапа:

1. Сначала выполняется транскрибация звонка через Whisper.
2. Затем транскрипт вместе со сценарием анализа и индивидуальными метриками передается в LLM для structured AI-анализа.

Важный момент: backend не пересчитывает смысловые оценки по собственной формуле. Он в основном:

- задает модели требуемую структуру ответа;
- нормализует значения;
- локализует ответ на русский язык при необходимости;
- сохраняет результат;
- агрегирует сохраненные оценки в отчетах и дашборде.

Это значит, что сами поля `Сентимент`, `Риск`, `Score` и `Соблюдение скрипта` появляются из structured AI-ответа модели, а не из жестко зашитой математической формулы в backend.

## Какие поля запрашиваются у модели

Backend ожидает от AI-анализа JSON со следующими ключевыми блоками:

- `overview.sentiment`
- `overview.riskLevel`
- `summary`
- `recommendations`
- `scriptAnalysis.overallScore`
- `scriptAnalysis.compliancePercent`
- `scriptAnalysis.checkpoints`
- `customMetrics`
- `nextStep`

Если structured AI-анализ вернулся пустым или некорректным, транскрипт может сохраниться отдельно как частичный результат, но итоговые оценки разговора будут отсутствовать.

## Сентимент

### Что это такое

`Сентимент` показывает эмоциональную окраску разговора в целом.

В текущей модели используются категории:

- `Позитивный`
- `Нейтральный`
- `Негативный`
- `Смешанный`

### Как формируется

Эти значения возвращаются моделью в поле:

```json
{
  "overview": {
    "sentiment": "Позитивный"
  }
}
```

После этого backend нормализует возможные варианты записи:

- английские значения `positive`, `neutral`, `negative`, `mixed`;
- русские значения в ожидаемый канонический вид.

### Настраивается ли

Сейчас отдельно не настраивается.

То есть в интерфейсе нет:

- пользовательских правил сентимента;
- весов;
- словарей тональности;
- порогов переключения между категориями.

На практике сентимент косвенно зависит от:

- качества транскрипта;
- текста сценария анализа;
- общего промпта AI-анализа.

## Риск

### Что это такое

`Риск` показывает вероятность проблемного исхода звонка с точки зрения разговора:

- потеря клиента;
- конфликт;
- неудовлетворенность;
- нерешенный вопрос;
- слабый следующий шаг;
- иные признаки проблемного контакта.

В текущей модели используются категории:

- `Низкий`
- `Средний`
- `Высокий`

### Как формируется

Значение возвращается моделью в поле:

```json
{
  "overview": {
    "riskLevel": "Высокий"
  }
}
```

Backend нормализует записи вида:

- `low`, `medium`, `high`;
- русские варианты в канонический формат.

### Настраивается ли

Сейчас отдельно не настраивается.

В интерфейсе пока нет:

- правил расчета уровня риска;
- набора риск-факторов;
- пользовательских порогов перехода между уровнями.

Риск также определяется моделью по содержанию транскрипта и контексту сценария.

## Score

### Что это такое

`Score` в интерфейсе и отчетах соответствует полю:

`scriptAnalysis.overallScore`

Это общая оценка качества разговора по сценарию.

### Как формируется

Значение возвращается моделью в structured AI-ответе:

```json
{
  "scriptAnalysis": {
    "overallScore": 78
  }
}
```

Backend:

- принимает число как есть;
- проверяет, что это корректное numeric-значение;
- сохраняет его;
- использует в отчетах, в таблице звонков и на дашборде.

Средний `score` на дашборде считается уже самим backend как среднее арифметическое по всем сохраненным анализам, где `overallScore` существует.

### Настраивается ли

Напрямую нет.

Сейчас не вынесены в настройки:

- шкала score;
- веса отдельных этапов скрипта;
- формула пересчета checkpoint'ов в итоговый балл;
- пороги для "хорошо / плохо".

Но `Score` косвенно настраивается через сам сценарий анализа:

- чем подробнее описан сценарий;
- чем точнее перечислены ожидаемые этапы разговора;
- чем конкретнее сформулированы критерии проверки,

тем стабильнее модель возвращает осмысленный `overallScore`.

## Соблюдение скрипта

### Что это такое

`Соблюдение скрипта` в UI соответствует полю:

`scriptAnalysis.compliancePercent`

Это процент соответствия звонка выбранному сценарию.

### Как формируется

Модель возвращает:

```json
{
  "scriptAnalysis": {
    "compliancePercent": 65,
    "checkpoints": [
      {
        "name": "Приветствие",
        "status": "Пройден",
        "comment": "Менеджер представился и обозначил цель звонка"
      }
    ]
  }
}
```

Помимо самого процента, модель возвращает и детализацию:

- `strengths`
- `violations`
- `checkpoints`

Именно эти checkpoint'ы дают расшифровку, почему процент такой, а не другой.

### Настраивается ли

Да, но не как отдельная формула, а через сценарий анализа.

На соблюдение скрипта влияют:

- поле сценария с описанием скрипта;
- полнота и качество этапов, которые нужно проверять;
- индивидуальные метрики, если они добавлены в сценарий.

Если сценарий слишком общий, `compliancePercent` тоже получается слишком условным.

Если сценарий подробно описывает:

- обязательное приветствие;
- выявление потребности;
- презентацию решения;
- работу с возражениями;
- фиксацию следующего шага,

то модель лучше оценивает соблюдение скрипта и формирует более полезные checkpoint'ы.

## Что реально настраивается сейчас

Сейчас пользователь может настраивать оценку разговора косвенно через сценарии.

Настраиваемые элементы:

- текст сценария анализа;
- библиотека сценариев;
- сценарий по умолчанию;
- правила автоподбора сценария;
- индивидуальные метрики в составе сценария.

Через это можно влиять на:

- что считается правильным разговором;
- какие этапы разговора считать обязательными;
- какие дополнительные критерии проверять;
- какой сценарий будет применяться к конкретному звонку.

## Что пока не вынесено в отдельные настройки

Пока в приложении нет отдельных UI-настроек для:

- весов критериев;
- пользовательской формулы score;
- формулы риска;
- словаря сентимента;
- числовых порогов для compliance;
- отдельных коэффициентов для checkpoint'ов.

Если понадобится более детерминированная оценка, следующий шаг по архитектуре должен быть таким:

1. Ввести формальную модель критериев в сценарии.
2. Хранить веса checkpoint'ов в БД.
3. Разделить AI-этап "извлечение фактов" и backend-этап "математический расчет итоговой оценки".

Тогда оценка станет менее зависимой от свободного вывода модели и более предсказуемой.

## Как это отражается в отчете

В отчете и на дашборде:

- `Проанализировано звонков` — количество сохраненных анализов;
- `Средний score` — среднее арифметическое по `scriptAnalysis.overallScore`;
- `Высокий риск` — количество звонков, где `overview.riskLevel = Высокий`;
- `Позитив / нейтрал / негатив` — распределение по `overview.sentiment`.

То есть KPI в отчете не рассчитываются заново по транскрипту, а собираются поверх уже сохраненных результатов анализа.

## Практический вывод

Если отвечать коротко на вопрос "настраивается ли эта оценка", то:

- `Сентимент` — напрямую не настраивается;
- `Риск` — напрямую не настраивается;
- `Score` — напрямую не настраивается формулой, но зависит от сценария;
- `Соблюдение скрипта` — настраивается через сценарий анализа и его детализацию.

Для текущей архитектуры главный инструмент настройки качества оценки — это хорошо проработанный сценарий анализа.

Практическая инструкция по заполнению сценария вынесена отдельно:
