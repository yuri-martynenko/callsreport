# Архитектура

## Назначение документа

Этот документ описывает общую архитектуру приложения: из каких блоков оно состоит, как они взаимодействуют и где проходит граница между интерфейсом, backend, внешними API и хранилищем.

## Общая схема

```text
Пользователь
  -> Web UI (public/index.html, public/app.js, public/styles.css)
  -> Backend API (server.js)
  -> VibeCode API
     -> activities
     -> files/download
     -> ai/audio/transcriptions
     -> ai/chat/completions
  -> Bitrix24 CRM ссылки
  -> Постоянная БД (db.js, APP_DB_PATH)
```

## Основные слои

### 1. Frontend

Frontend находится в каталоге `public/` и отвечает за:

- загрузку списка звонков;
- отображение дашборда;
- показ детализации анализа;
- редактирование сценариев;
- изменение настроек автотранскрибации;
- запуск ручного анализа;
- отображение статусов очереди анализа.

Frontend не содержит бизнес-логики анализа. Он только запрашивает данные у backend и отображает результат.

### 2. Backend

Backend реализован в `server.js` и отвечает за:

- HTTP API для frontend;
- интеграцию с VibeCode API;
- нормализацию данных звонков;
- управление очередью анализа;
- выбор сценария анализа;
- запуск транскрибации и AI-анализа;
- сохранение результатов в БД;
- выдачу агрегированных данных для отчета и дашборда.

При старте backend сначала выполняет единый env-bootstrap через `env.js`, а уже потом загружает остальные слои. Это сделано для того, чтобы `server.js` и `db.js` работали с одним и тем же набором переменных окружения и одинаково вели себя в local/dev и production runtime.

### 3. Слой хранения

Слой хранения вынесен в `db.js`.

Он отвечает за:

- инициализацию БД;
- создание схемы;
- миграцию legacy JSON-данных;
- чтение и запись аналитики;
- чтение и запись сценариев;
- чтение и запись настроек;
- сериализацию изменений в файл БД.

`db.js` не реализует собственную загрузку `.env`: он использует тот же общий bootstrap, что и backend API слой.

Дополнительно в SQLite хранится локальный кэш CRM-клиентов (`crm_client_cache`), который используется для ускорения загрузки списка звонков. Backend больше не делает массовую синхронную догрузку CRM-сущностей при каждом `GET /api/calls`: сначала отдаётся уже известное имя клиента из звонка или локального кэша, а недостающие имена дозаполняются в фоне и сохраняются для следующих открытий приложения.

### 4. Внешние сервисы

Система использует:

- `VibeCode API` как основной backend для:
  - получения звонков;
  - скачивания записей;
  - транскрибации аудио;
  - текстового AI-анализа;
- `Bitrix24` как источник бизнес-данных и CRM-контекстов.

## Основные пользовательские разделы

### Дашборд

Показывает агрегированные показатели:

- количество проанализированных звонков;
- средний score;
- высокий риск;
- распределение по тональности;
- графики по менеджерам и дням.

### Отчет

Показывает:

- фильтруемый список звонков;
- текущий статус анализа по каждому звонку;
- выбранный сценарий;
- детализацию по выбранной строке.

### Сценарии анализа

Раздел для управления сценариями:

- создание;
- редактирование;
- удаление;
- настройка правил автоподбора;
- назначение сценария по умолчанию.

### Настройки

Содержит настройки поведения системы, в первую очередь:

- режим автоматической расшифровки звонков.

## Очередь анализа

Очередь реализована внутри backend, без внешнего брокера.

Есть два режима постановки в очередь:

- ручной запуск из интерфейса;
- автоматическая постановка по настройке автотранскрибации.

Очередь обрабатывается фоновым циклом с ограничением конкурентности:

- `ANALYSIS_QUEUE_CONCURRENCY`
- `ANALYSIS_AUTO_SCAN_INTERVAL_MS`
- `ANALYSIS_AUTO_SCAN_BATCH`

## Принцип разделения ответственности

- `public/*` отвечает только за UI и вызов API;
- `server.js` отвечает за бизнес-логику;
- `db.js` отвечает за постоянное хранение;
- `docs/*` отвечает за фиксацию решений и правил сопровождения.

## Архитектурные ограничения

- текущая очередь встроенная, без Redis/RabbitMQ;
- БД файловая, поэтому приложение предполагает один активный экземпляр сервиса;
- сильная зависимость от формата и стабильности VibeCode API;
- качество анализа зависит от доступности Whisper и LLM-провайдера.
